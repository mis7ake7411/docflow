# 使用者管理與新增使用者（實作計畫）

## Constitution Check（前）
- 規格是否避免實作細節：通過
- 是否以 MVP 為最小切片：通過
- 使用者故事是否可獨立驗收：通過
- 成功準則是否可量測且技術無關：通過
- 釐清事項是否 ≤ 3：通過
- 是否拒絕過度設計：通過

## 實作步驟（高階）
1. 角色與權限定義
   - 新增 `MANAGER` 角色與其權限範圍（使用者管理 + 既有資料查看）。
   - 更新權限矩陣，確保 `MANAGER` 可新增使用者、可查看既有資料，但不可新增/修改/刪除文件與資料夾。
2. 使用者管理能力
   - 提供使用者列表與單一關鍵字搜尋（username/email）。
   - 提供管理端新增使用者（`ADMIN`/`MANAGER`，可設定角色/狀態），並產生臨時密碼回傳顯示。
   - 提供使用者角色/狀態更新與啟用/停用。
3. 首次登入改密碼
   - 建立「臨時密碼」狀態與驗證機制（至少 8 碼，需包含英文與數字）。
   - 登入後若需改密碼，阻擋其他功能並引導完成改密碼。
4. 自助註冊
   - 登入頁提供註冊入口。
   - 註冊完成後預設 `USER + ACTIVE`。
5. 前端使用者管理介面
   - 列表欄位：帳號、Email、角色、狀態、建立時間。
   - 功能：搜尋、新增、編輯（角色/狀態）、啟用/停用。
6. 權限驗證與拒絕行為
   - 未授權角色不可見入口，且存取行為需被拒絕並提示。

## 複雜度追蹤表
| 複雜度來源 | 原因 | 決策 | 拒絕的替代方案與理由 |
| --- | --- | --- | --- |
| 臨時密碼 + 首次登入改密碼 | 需額外狀態與阻擋流程 | 保留（MVP 核心需求） | 不做強制改密碼：降低安全性且不符合需求 |
| 新增 `MANAGER` 角色 | 需擴充角色與權限矩陣 | 保留（已確認需求） | 僅 ADMIN：不符合使用者需求 |
| 郵件邀請/重設密碼 | 需外部服務與流程 | 拒絕（超出 MVP） | 暫緩至有郵件服務再評估 |

## 驗收與測試重點
- 驗收情境依規格列出逐項確認。
- 角色權限邊界（`ADMIN` / `MANAGER` / `USER`）需可驗證。
- 首次登入改密碼需可驗證「未改密碼不可使用其他功能」。
